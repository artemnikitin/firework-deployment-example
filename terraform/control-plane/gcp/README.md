# GCP control plane

This stack deploys the Firework control-plane on **GKE Autopilot**. Three roles (events, registry, controller) run as Kubernetes Deployments in a regional cluster. Secrets are delivered via the Secrets Store CSI driver from Secret Manager. State and rendered node configs are stored in GCS via native ADC/Workload Identity — no HMAC keys or S3-shim.

Terraform state is **local** (no remote backend). The state file lives at `terraform.tfstate` in this directory.

---

## Prerequisites

### 1. Bootstrap APIs (once per project)

Terraform cannot manage Google APIs until Service Usage and Cloud Resource Manager are themselves enabled. Run once before the first `apply`:

```bash
gcloud services enable \
  serviceusage.googleapis.com cloudresourcemanager.googleapis.com \
  --project=YOUR_GCP_PROJECT
```

All other required APIs (`container.googleapis.com`, `secretmanager.googleapis.com`, etc.) are enabled by the `google_project_service.required` resources in `services.tf` on the first apply.

### 2. Cloud DNS managed zone

The stack expects a pre-existing Cloud DNS managed zone whose name matches `var.dns_zone_name` (default `firework-gcp`). Create it once and delegate from your parent zone:

```bash
gcloud dns managed-zones create firework-gcp \
  --dns-name="gcp.example.com." \
  --description="Firework GCP zone" \
  --project=YOUR_GCP_PROJECT

# Copy the NS records it prints into your parent zone / dns-foundation stack.
```

Verify delegation: `dig NS gcp.example.com +short`

### 3. Terraform identity permissions

The principal running `apply` needs at minimum:

| Role | Why |
|---|---|
| `roles/container.admin` | Create GKE Autopilot cluster |
| `roles/compute.networkAdmin` | VPC, subnets, IPs, NAT |
| `roles/iam.serviceAccountAdmin` | Create runtime service account |
| `roles/iam.workloadIdentityPoolAdmin` | WI binding on the service account |
| `roles/secretmanager.admin` | Grant `secretAccessor` on each secret |
| `roles/storage.admin` | Create state/config bucket, grant IAM |
| `roles/dns.admin` | DNS A records |
| `roles/logging.configWriter` | Log metric filters |
| `roles/monitoring.editor` | Log metrics |
| `roles/resourcemanager.projectIamAdmin` | Project-level `logWriter`/`metricWriter` grants |

**`projectIamAdmin` is mandatory.** The stack grants project-level `roles/logging.logWriter` and `roles/monitoring.metricWriter` to the runtime service account. Without `projectIamAdmin`, those `google_project_iam_member` resources fail with `403 The caller does not have permission` even though all other resource-scoped IAM bindings succeed.

> **Tip:** If `GOOGLE_IMPERSONATE_SERVICE_ACCOUNT` or `GOOGLE_APPLICATION_CREDENTIALS` is set in your shell, Terraform runs as that service account, not your gcloud user. Verify the effective identity with `gcloud auth list` in the apply shell.

### 4. Pre-create the webhook secret

Only the GitHub webhook secret must exist before the first `apply`. All TLS certs, the enrollment CA, and the bootstrap token are **auto-generated by Terraform** when `auto_create_demo_secrets = true` (the default).

```bash
PROJECT=YOUR_GCP_PROJECT

# GitHub webhook secret — generate a random string and set the same value in the GitHub webhook UI.
WEBHOOK_SECRET=$(openssl rand -hex 32)
echo -n "$WEBHOOK_SECRET" | gcloud secrets create firework-webhook-secret \
  --data-file=- --replication-policy=automatic --project=$PROJECT
```

Terraform will create and populate the following secrets automatically on first apply:
- `firework-cp-enrollment-ca-cert` / `firework-cp-enrollment-ca-key` — enrollment CA
- `firework-cp-events-tls-cert` / `firework-cp-events-tls-key` — events server TLS
- `firework-cp-registry-tls-cert` / `firework-cp-registry-tls-key` — registry server TLS
- `firework-cp-bootstrap-token` — node self-enrollment token

To bring your own certs instead, set `auto_create_demo_secrets = false` and supply all `*_secret_id` variables pointing to your pre-created secrets.

### 5. Container image

The control-plane runs from a container image. Set `var.controlplane_image` to a published image:

```
controlplane_image = "ghcr.io/artemnikitin/firework-controlplane:v0.1.0"
```

## Deploy

```bash
cd terraform/control-plane/gcp
cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars — fill in gcp_project, events_domain, state_bucket_name,
# controlplane_image, git_repo_url, and webhook_secret_id.
terraform init
terraform apply
```

Key outputs you'll need for the data-plane:

```bash
terraform output config_bucket_name          # pass as data-plane config_bucket_name
terraform output registry_url               # pass as data-plane registry_url
terraform output registry_server_name       # pass as data-plane registry_server_name
terraform output registry_ca_secret_id      # pass as data-plane registry_ca_secret_id
terraform output registry_bootstrap_token_secret_id
terraform output events_webhook_url         # configure as GitHub webhook target
```

If `use_control_plane_remote_state = true` in the data-plane (the default), it reads these from `../../control-plane/gcp/terraform.tfstate` automatically — no manual copying needed.

## Teardown

The state/config bucket has versioning enabled and `force_destroy` defaults to `false`. A plain `terraform destroy` fails unless `state_bucket_force_destroy = true` is set (it is in `terraform.tfvars`):

```bash
terraform destroy
```
